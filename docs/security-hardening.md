# Защита игровых комнат

Продакшен работает через GitHub Pages и Supabase, поэтому проверки состояния
партии должны выполняться в базе, а не только в браузере.

## Что включено

`supabase/schema.sql` и отдельная миграция
`supabase/harden-guest-rooms.sql` устанавливают `BEFORE UPDATE`-триггер для
`public.rooms`. Он отклоняет:

- изменение `game_state` без увеличения `game_version`;
- позиции, где у стороны не ровно 15 шашек;
- некорректные номера пунктов, цвета и количества шашек;
- победу без 15 снятых шашек и без подтверждённой сдачи, выхода или потери связи.

Локальный Node-сервер выполняет те же проверки и отклоняет устаревшую версию
состояния с HTTP 409.

## Применение

Для нового проекта выполните весь `supabase/schema.sql`. Для существующего
проекта достаточно один раз выполнить только
`supabase/harden-guest-rooms.sql` в Supabase SQL Editor.

После применения проверьте одну онлайн-партию, сдачу и завершение по потере
соединения.

## Ограничение

Проверка гарантирует структурную целостность состояния, но текущий клиент всё
ещё публикует готовую доску. Полная защита от изменённого клиента потребует
серверной генерации кубиков и RPC/Edge Function, которая принимает отдельные
ходы, проверяет их движком правил и только затем сохраняет состояние.
